<Phase type="reflection">

A good agent spec answers these questions **before** any code is written:

| Question | What goes in the answer |
|---|---|
| **Trigger** | What kicks off the agent? (cron / webhook / event / manual run) |
| **Inputs** | What arrives? (format, volume, source) |
| **Steps** | What nodes does the pipeline consist of? |
| **AI vs Code** | Which nodes are LLM, which are code (see u2)? |
| **Outputs** | Where does the result get written? |
| **Failure modes** | What can break, what to do? |
| **Observability** | Where do you look to see what happened? |
| **Cost ceiling** | Max spend per day? |
| **Human-in-loop** | Where is your review needed? |

If even one question has no answer — it's too early to write code.

</Phase>

<Phase type="concept">

**Agent specification template**

```markdown
# Agent: [name]

## Goal
[One sentence: what the agent does, what pain it solves]

## Trigger
- Type: cron / webhook / event / manual
- Params: [e.g., every 5 minutes / on Gmail label / on repo push]

## Pipeline
1. [Node type] — [what it does] — [in → out]
2. [Node type] — [what it does] — [in → out]
3. ...

## AI nodes
List of nodes where an LLM is used:
- Node X: model [provider/model], prompt in `prompts/x.md`
- ...

## Tools / APIs
- Gmail API (scope: ...)
- Notion API (workspace: ...)
- ...

## Storage
- Where the result is written
- Idempotency: how we avoid duplicates

## Observability
- Trace: Langfuse / OTel / stdout logs
- Alerting: where errors go (Telegram / email / Slack)

## Failure modes
| Scenario | Behavior |
|---|---|
| LLM 500 | Retry 3× with exp backoff, then fallback model |
| Bad input | Log + Telegram alert + skip |
| Storage full | Hard fail + alert |

## Cost
- LLM calls per day: ~N
- Estimated $: ~$X
- Hard cap: $Y (via provider budgets)

## Human-in-loop
- Where: [step N]
- What you confirm
- How: [Telegram message / Notion task / email]
```

**Real example of a filled-out spec** — a simplified version of ENERV ingest:

```markdown
# Agent: ENERV Gmail Ingest

## Goal
Emails tagged `AI Ingest` automatically land in the correct folder of an
Obsidian vault as .md files.

## Trigger
- Type: poll
- Params: Gmail API every minute, filter `label:AI Ingest`

## Pipeline
1. Gmail Trigger — fetch emails — () → [emails]
2. Normalize Input — standardize fields — [email] → [normalized]
3. Gemini Router — classification — [normalized] → [folder, filename]
4. Parse Response — JSON validation — [llm_out] → [structured]
5. ACL Transformer — generate .md — [normalized + structured] → [md]
6. Source Check — dedup — [md] → [md or skip]
7. Write — file write — [md] → /home/node/obsidian/{folder}/{filename}.md
8. Mark Ingested — Gmail label — [email_id] → ()
9. Remove Ingest Label — remove trigger — [email_id] → ()

## AI nodes
- Gemini Router — gemini-flash, prompt `gemini-router.md`
- ACL Transformer — gemini-flash, prompt `acl-transformer.md`

## Tools
- Gmail API (modify scope)
- File system (volume mount /home/node/obsidian)

## Storage
- Obsidian vault on host volume
- Idempotency: label `AI Ingest` is removed, `AI-Ingest-Done` is added

## Observability
- n8n executions log
- Langfuse traces for both LLM nodes

## Failure modes
| Scenario | Behavior |
|---|---|
| Gemini 500 | n8n retry 3× with exp backoff |
| Broken JSON | Log + email is left with `AI Ingest` |
| Disk full | Email alert (Resend) |

## Cost
- ~10 emails/day × 2 LLM calls = 20 calls/day
- Gemini Flash: ~$0.001/call → ~$0.02/day
- Hard cap: $5/month via Vertex AI budget

## Human-in-loop
- Emails tagged `AI-Ingest-Review` — once a week I review the folder
  Atlas/Notes/Sources manually to catch misclassifications
```

This is **on paper**. There's no code yet. But anyone — including me six months from now — can read it and understand what the agent does.

<Phase type="practice">

Fill out the spec template above for **your** task from unit 1.

Constraints:
- Don't write "as if I already know everything." If you don't know — write `[?]` and leave the question.
- One file `my-experiments/my-agent-spec.md`
- Minimum 1 page, maximum 3
- Show it to someone (partner, friend, ChatGPT) and ask: "is it clear what the agent does?"

**Sanity check before ending the unit:**

- [ ] Trigger is clear
- [ ] Pipeline has 3+ steps
- [ ] At least 1 LLM node and 1 non-LLM node
- [ ] At least 1 failure mode described
- [ ] Clear what it ~costs per day
- [ ] Specified where to look at traces

If all 6 boxes are ticked — you've got a spec you can start building from.
