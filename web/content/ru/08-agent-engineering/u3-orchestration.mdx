<Phase type="activation">

Прокрути в голове любой реальный workflow которым ты пользуешься каждую неделю — например, обработка inbox.

Не описывай это как «шаги которые надо выполнять» — представь это как **кукловода, ведущего разом несколько марионеток**: каждая нить в отдельной руке, каждая марионетка делает своё. Когда всё держит один человек — это искусство координации, не мускульная работа.

Разгляди свой workflow как такую систему: какие марионетки сейчас дёргаешь ты сам — и какие из них могли бы двигаться сами, если бы нить была натянута правильно?

</Phase>

<Phase type="reflection">

Идея оркестрации: вместо одной большой задачи у одного агента — **много маленьких задач у разных агентов** (или у разных шагов одного агента).

Преимущества:
- **Каждый шаг проще** → меньше шансов на ошибку
- **Каждый шаг тестируется отдельно** → проще отлаживать
- **Шаги переиспользуются** → один Gmail-trigger используется в 10 пайплайнах
- **Восстанавливаемость** → если шаг 4 сломался, шаги 1–3 не теряются

Не представляй это как оркестр — представь это как **ткацкий станок**: каждая нить идёт по своему ряду, переплетается с другими в точно заданный момент, и создаёт паттерн. Обрыв одной нити — не конец ткани: ты видишь где именно, находишь место, чинишь. Ткань до и после этого места остаётся целой.

Это **mindset shift**: ты больше не «пишешь промпт». Ты **проектируешь ткацкий станок**.

Поймай образ: где в твоём pipeline нити переплетаются — и в какой точке обрыв разрушит всю ткань, а не только локальный ряд?

</Phase>

<Phase type="concept">

**Реальный кейс: ENERV n8n pipeline**

ENERV — моя система personal knowledge management. Каждое письмо помеченное в Gmail тегом `AI Ingest` автоматически разбирается и попадает в правильную папку Obsidian-vault'а (1433 файла на данный момент).

**Архитектура pipeline:**

```
┌─────────────────────────────────────────────────────┐
│  Gmail Trigger (polls every 1 min, label:AI Ingest) │
└──────────────────────┬──────────────────────────────┘
                       ↓
┌─────────────────────────────────────────────────────┐
│  Normalize Input  (JS code: standardize fields)     │
└──────────────────────┬──────────────────────────────┘
                       ↓
┌─────────────────────────────────────────────────────┐
│  Gemini Router  (LLM: classify into target folder)  │
│  → returns: {folder, filename, tags}                │
└──────────────────────┬──────────────────────────────┘
                       ↓
┌─────────────────────────────────────────────────────┐
│  Parse Response  (JS code: validate JSON)           │
└──────────────────────┬──────────────────────────────┘
                       ↓
┌─────────────────────────────────────────────────────┐
│  ACL Transformer  (LLM: write structured .md)       │
└──────────────────────┬──────────────────────────────┘
                       ↓
┌─────────────────────────────────────────────────────┐
│  Source Check  (Code: avoid duplicates)             │
└──────────────────────┬──────────────────────────────┘
                       ↓
┌─────────────────────────────────────────────────────┐
│  Write to /home/node/obsidian/{folder}/{name}.md    │
└──────────────────────┬──────────────────────────────┘
                       ↓
┌─────────────────────────────────────────────────────┐
│  Mark Ingested  (Gmail: add Label_22)               │
└──────────────────────┬──────────────────────────────┘
                       ↓
┌─────────────────────────────────────────────────────┐
│  Remove Ingest Label  (Gmail: prevent re-process)   │
└─────────────────────────────────────────────────────┘
```

**Что важно заметить:**

1. **AI только в 2 узлах из 9** — Gemini Router (классификация) и ACL Transformer (написание текста). Остальное — детерминированный код или вызовы API. Это применение jagged intelligence.

2. **Idempotency через labels** — после успешной обработки письмо получает `AI-Ingest-Done` и теряет `AI Ingest`. Если пайплайн упадёт и перезапустится, обработанные письма не схватятся повторно.

3. **Trigger + Action разделены** — Gmail-trigger дёргается n8n'ом раз в минуту независимо. Любая ошибка в action не ломает trigger.

4. **Структура vault'а — часть spec'а** — Gemini знает что есть папки `Atlas/Notes/Sources`, `Efforts/On`, `Calendar/Logs`, `Facets`. Это **constraint** который вкладывается в системный промпт.

**Это не «один большой промпт». Это спроектированная архитектура.**

</Phase>

<Phase type="practice">

Возьми свой workflow из активационной фазы (Gmail inbox или любой другой). Нарисуй его как блок-схему по образцу ENERV. Для каждого узла укажи:

- Тип: `Trigger`, `Code`, `LLM`, `Tool`, `Write`
- Что узел получает на вход
- Что отдаёт на выход
- Как обрабатывает ошибку (retry / fail / human)

Используй ASCII-блоки как в концепте, или нарисуй на бумаге и сфотографируй.

Сохрани в `my-experiments/u3-orchestration-blueprint.md`.

**Проверь себя:**
- Есть ли LLM-узел где можно обойтись кодом? (упрости)
- Есть ли узел где код пытается делать то, что лучше делает LLM? (переделай)
- Есть ли точка где ошибка приведёт к потере данных? (добавь idempotency)

</Phase>
