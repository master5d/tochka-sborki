<Phase type="activation">

Ты прошёл через 4 unit'а и сделал выбор стека. Этот последний unit — **твой план миграции** на 7 дней.

Цель: не «всё сразу», а постепенно перейти с того что у тебя сейчас → на выбранный стек, чтобы реальная работа не остановилась.

</Phase>

<Phase type="reflection">

Самая частая ошибка миграции: «удалю Claude Code, поставлю Aider, и сразу буду эффективнее». Не работает. Новый стек требует **другого мышления** (особенно SOVERN с boss-worker).

Реалистичный путь:
- **Дни 1–2:** новый стек как **дополнение**. Старый продолжает работать.
- **Дни 3–4:** перенос **одной** реальной задачи на новый стек. Засекаешь, насколько медленнее / быстрее.
- **Дни 5–7:** решаешь — оставить hybrid или полностью мигрировать.

</Phase>

<Phase type="concept">

## Шаблон плана миграции

```markdown
# Миграция на [твой стек] — 7 дней

## День 1: Setup
- [ ] Установить Frontend agent (Hermes / Aider / Cline / Claude Code)
- [ ] Поднять LiteLLM gateway локально
- [ ] Получить API keys (Cerebras / Gemini / etc.)
- [ ] Сделать «hello world» запрос через curl

## День 2: Reproduce
- [ ] Взять ОДНУ задачу, которую ты решил на старом стеке за последнюю неделю
- [ ] Повторить её на новом стеке
- [ ] Записать: где медленнее / где быстрее / где проблема

## День 3–4: Real work
- [ ] Сделать одну реальную рабочую задачу на новом стеке
- [ ] Старый стек оставить как fallback
- [ ] Не уговаривай себя что «новое лучше» — фиксируй факты

## День 5: Customize
- [ ] Настрой 1–2 skill / hook под свой workflow
- [ ] Добавь свой CLAUDE.md / AGENTS.md для нового агента
- [ ] LiteLLM: настрой fallback chain

## День 6: Honest comparison
- [ ] Запиши: что лучше на новом стеке, что хуже
- [ ] Где hybrid имеет смысл (например, Claude Code для дизайна + Aider для рефакторинга)

## День 7: Decision
- [ ] Решение: полная миграция / hybrid / возврат
- [ ] Если возврат — что нужно от стека чтобы передумать?
- [ ] Если миграция — закрепляй: убираешь старый stack из активной памяти
```

</Phase>
