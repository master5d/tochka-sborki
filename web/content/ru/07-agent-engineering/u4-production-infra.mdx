<Phase type="activation">

Прототип агента который работает на твоём ноутбуке — это **20% работы**.

Чтобы этот же агент работал когда:
- Ты спишь
- Ноут закрыт
- Интернет провайдер моргнул
- LLM-провайдер уронил endpoint
- Один из шагов вдруг стоит $0.10 вместо $0.001
- Кто-то прислал письмо с эмодзи которое ломает парсер

— нужны остальные 80%.

Запиши: что в твоей задаче из юнита 1 произойдёт, если **в момент исполнения**:
- LLM провайдер вернёт 500 в течение 30 секунд?
- Один из шагов потребует 5 минут вместо 5 секунд?
- Тебя нет за компом и ты не увидишь ошибку?

Это вопросы которые **отделяют прототип от продакшена**.

</Phase>

<Phase type="reflection">

Production-инфраструктура агентских систем включает 5 блоков:

1. **Execution** — где и как запускается
2. **Observability** — как ты видишь что происходит
3. **Reliability** — что делать при ошибке
4. **Cost control** — как не разориться
5. **Security** — кто имеет доступ

Vibe-coder часто думает только про (1). Agent engineer думает про все пять.

Каждый блок — это **накопленный опыт чужих факапов**. Дальше — реальные грабли на моей собственной инфре SOVERN (Hetzner CX22 + Cloudflare + Docker), на которой крутится ENERV.

</Phase>

<Phase type="concept">

**SOVERN — production stack для агентских пайплайнов**

Конфигурация:
- **Server:** Hetzner CX22 (Ubuntu, ~4GB RAM, ~€5/мес)
- **Network edge:** Cloudflare Tunnel — публичные домены без открытых портов
- **Containers:** n8n (оркестрация), Langfuse v2 (трейсы), Postgres, cloudflared
- **Bridge network:** `sovern-net` — внутренние имена `enerv-n8n`, `enerv-logs`

**Грабли которые стоили мне дней отладки** (сохрани, проходить через них необязательно):

**Грабли 1: Двойной cloudflared = 502 flapping**
> Поднял cloudflared и через systemd на хосте, и контейнером в Docker — с одним и тем же tunnel token. CF балансировал между ними, хостовый не видел Docker DNS (systemd-resolved не знает про bridge), получались случайные 502.
> **Правило:** один tunnel — один процесс. Держи cloudflared **внутри** той же Docker-сети что и origins.

**Грабли 2: Публичный IP как origin тоннеля**
> Маршрутизировал `n8n.synergify.com` → `91.99.62.63:5678`. Это hairpin через публичную сеть — лишний hop, лишняя задержка, потенциальные таймауты.
> **Правило:** в tunnel-конфиге origin = Docker hostname (`http://enerv-n8n:5678`) или `host.docker.internal`.

**Грабли 3: Langfuse v3 без ClickHouse**
> Langfuse v3 требует ClickHouse + Redis (~2GB лишних RAM). На CX22 это смерть. Пинуй v2 тег явно.
> **Правило:** читай release notes требований к ресурсам **до** обновления.

**Грабли 4: n8n API не работает извне**
> `curl https://n8n.synergify.com/api/v1` снаружи возвращает 200, но Python `urllib` получает 403 (видимо, CF challenge headers). Скрипты которые дёргают n8n API нужно запускать **на самом сервере**, обращаясь к `http://localhost:5678/api/v1`.
> **Правило:** для управления — SSH + локальный URL. Публичный URL — только для UI.

**Helper для длительных скриптов:**

```bash
# /usr/local/bin/run_bg на сервере
run_bg /tmp/my-script.py
# Запускает через nohup, отрывается от SSH сессии
# Логи: /root/sovern/logs/my-script_YYYYMMDD_HHMMSS.log
# Возвращает PID и путь к логу
```

Это решает «закрыл SSH — скрипт умер». Любой скрипт >30 секунд — через `run_bg`.

**Pattern: 5-fallback LLM routing**

В LiteLLM конфиге для моего стека:

```yaml
model_list:
  - anthropic/claude-sonnet      # primary
  - openai/gpt-4o                # fallback 1
  - google/gemini-flash          # fallback 2
  - groq/llama-3-70b             # fallback 3 (fast/cheap)
  - openai/router.huggingface.co # fallback 4 (free)
```

Когда первый провайдер ложится — следующий подхватывает. Это **reliability** на уровне инфры, не на уровне промптов.

**Observability через Langfuse:**

Каждый LLM-вызов в моих агентах автоматически уходит в Langfuse: input, output, цена, латентность, какая модель сработала, какая упала. Через две недели можно зайти и посмотреть **что именно AI ответил** на конкретном письме в 03:47 ночи.

Без этого — ты слепой инженер.

</Phase>

<Phase type="practice">

Для твоего workflow из юнита 3 ответь:

**Execution:**
- Где он запускается? (твой ноут? серверный cron? n8n trigger?)
- Что если запуск выпадет на момент когда машина выключена?

**Observability:**
- Если шаг 4 ошибся 3 дня назад — как ты узнаешь?
- Как ты увидишь что именно LLM ответил?

**Reliability:**
- Что делать если LLM-провайдер дал 500?
- Что если входные данные пришли в неожиданном формате (битый JSON, эмодзи в имени файла)?

**Cost:**
- Сколько LLM-вызовов в день у тебя в худшем случае?
- Что произойдёт если сумма пойдёт умножаться (например, retry-цикл)?

**Security:**
- Где хранятся API-ключи?
- Что увидит другой человек если получит доступ к ноуту/серверу?

Это **production readiness checklist**. Сохрани ответы в `my-experiments/u4-readiness.md`.

Если на каждый вопрос ты ответил «не знаю / не подумал» — это не провал, это **сейчас стоит подумать**. Лучше сейчас на бумаге, чем в 2 ночи в производственном инциденте.

</Phase>
