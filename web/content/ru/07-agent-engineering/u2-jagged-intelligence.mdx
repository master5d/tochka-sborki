<Phase type="activation">

Известный факт: Claude/GPT может отрефакторить 100 000 строк кода — и **ошибиться при подсчёте букв «r» в слове "strawberry"**.

Это не баг. Это фундаментальное свойство современных LLM, которое называется **Jagged Intelligence** — рваный, неровный интеллект.

Подумай и запиши:
- Где AI тебя удивил мощностью?
- Где AI тебя удивил тупостью?

Эти два списка — твоя личная карта jagged intelligence для твоих задач.

</Phase>

<Phase type="reflection">

**Jagged Intelligence** = модель блестит в одних задачах и проваливается в других, причём логика «легко/сложно» **не совпадает** с человеческой.

Что AI делает легко:
- Переписать большой текст в другом стиле
- Найти паттерн в логах
- Сгенерировать boilerplate-код
- Просуммировать встречу
- Перевести между языками

Что AI делает плохо:
- Точный подсчёт (символы, числа, индексы массивов)
- Хронология событий ("что было раньше — А или Б")
- Геометрия и пространственное мышление
- Доступ к свежим данным без инструментов
- Удержание длинных контекстов без потерь

Это не временная проблема. Это **архитектурное** свойство трансформеров.

**Вывод для агентского инжиниринга:** ты не «делегируешь задачу AI». Ты **разбиваешь задачу на части**:
- Те, что AI делает супер → отдаёшь
- Те, что AI делает плохо → даёшь ему **инструмент** (калькулятор, БД, поиск)
- Те, что критичны → **верифицируешь** второй моделью или человеком

</Phase>

<Phase type="concept">

**Карта делегирования** (Jagged Intelligence Literacy):

```
┌─────────────────────────────────────────────────┐
│  AI отлично     │ AI с инструментом │ Человек   │
├─────────────────┼───────────────────┼───────────┤
│ Текст-в-текст   │ Точный счёт       │ Понимание │
│ Стилизация      │ Свежие данные     │ Вкус      │
│ Шаблоны кода    │ Долгая память     │ Этика     │
│ Резюме          │ Точные расчёты    │ Что и     │
│ Перевод         │ Внешние API       │ зачем     │
└─────────────────┴───────────────────┴───────────┘
```

**Правило вкуса для agent engineer:**

> *Поручай AI исполнение. Оставляй за собой направление и проверку.*

Это означает:
- Ты выбираешь — **что строить**
- AI пишет — **код, который это делает**
- Ты проверяешь — **что результат соответствует замыслу**
- AI чинит — **то, что ты подсветил как ошибку**

**Реальный пример: ENERV пайплайн**

В ENERV (моя система semantic journal) Gmail-письма с тегом `AI Ingest` обрабатываются так:

```
Gmail Trigger          ← инструмент, не AI
  ↓
Normalize Input        ← код, детерминирован
  ↓
Gemini Router          ← AI решает: куда положить (Atlas? Calendar? Facets?)
  ↓
Parse Response         ← код, детерминирован
  ↓
ACL Transformer        ← AI пишет .md в правильную папку
  ↓
Mark Ingested          ← инструмент (Gmail API)
```

AI здесь ровно в **двух точках** где он силён: классификация + текст-в-структуру. Всё остальное — детерминированные шаги.

Это и есть agent engineering — **спроектированная** комбинация AI-узлов и не-AI-узлов.

</Phase>

<Phase type="practice">

Возьми свою задачу из юнита 1. Разбей её на шаги. Для каждого шага классифицируй:

| Шаг | Тип | Кто делает |
|-----|-----|-----------|
| Скачать письма | Tool | Gmail API |
| Понять тему | AI | LLM |
| Записать в файл | Tool | File system |
| Проверить что не дубликат | Code | детерминированный код |
| Уведомить тебя | Tool | Telegram/Slack API |

Сохрани в `my-experiments/agent-decomposition.md` как таблицу.

**Главный вопрос:** для каких шагов ты прямо сейчас пытаешься использовать AI там, где **достаточно простого кода или API**? Это самый частый источник ненадёжности.

</Phase>
