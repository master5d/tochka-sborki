<Phase type="activation">

Представь двух дирижёров. Один получает перед концертом **партитуру** — с каждой нотой, темпом, динамикой, паузами. Другой слышит лишь: «Сыграй что-нибудь красивое».

Прокрути в голове: оба оркестра выходят на сцену. Что происходит дальше?

Теперь поймай момент: твой последний промпт AI — он был партитурой или «чем-нибудь красивым»?

</Phase>

<Phase type="reflection">

Удержи в голове образ **архитектора перед чертёжной доской**.

У него на столе — либо технический план с размерами, материалами и нагрузками, либо записка «сделай покрасивее». Разгляди оба варианта: что архитектор может построить в каждом случае?

Теперь поймай разницу: где в твоём промпте был план, а где — пожелание? Что из этого управляет результатом — а что отдаёт его на волю случая?

</Phase>

<Phase type="concept">

## Структура профессионального промпта: CTID

### Быстрая мнемоника для старта

```
Контекст   = кто ты и что за ситуация
Задача     = что конкретно нужно сделать
Инструкции = как именно, каким тоном, в каком формате
Данные     = примеры, ссылки, вводная информация
```

### Полная структура для воспроизводимых результатов

```markdown
## Роль
Ты — [экспертиза и специализация]

## Задача
[Конкретная цель — одно предложение]

## Входные данные
[Что ты предоставляешь: файлы, URL, данные, контекст]

## Ожидаемый результат
[Формат, структура, объём, критерии качества]

## Ограничения
- Не делать: [X]
- Использовать только: [Y]
- Максимум: [Z]
```

### Пример в действии

```markdown
## Роль
Ты — исследовательский агент, специализирующийся на анализе
технических документов и извлечении ключевых инсайтов.

## Задача
Проанализируй документацию по API и создай краткую справку
с примерами для Python-разработчиков среднего уровня.

## Ожидаемый результат
Markdown-файл со структурой:
1. Описание API (2–3 предложения)
2. Основные endpoints (таблица)
3. Authentication
4. 3 практических примера с кодом

## Ограничения
- Примеры должны быть полные (можно скопировать и запустить)
- Объём: не более 500 строк
- Используй Python 3.10+
```

> Чем точнее ТЗ — тем меньше итераций. Инвестиция в промпт экономит время в разы.

### Барьер — не в воображении

Принято думать, что людей останавливает «нет воображения». Это не так: образ результата есть почти у всех — сотни идей, которые годами не доходят до дела. Останавливает другое — **воображение не переведено в написанную задачу**. AI этот навык не отменяет; он его *вознаграждает*. Отсюда — три ступени, и порядок важен:

1. **Воображение** — у тебя есть образ результата. Он всегда был.
2. **Задача** — ты учишься *писать* этот образ как ТЗ (CTID выше) и выполнять его **вместе с AI**, оставаясь в цикле: проверяешь, правишь, снова просишь.
3. **Автоматизация** — и только когда писать и проверять задачу получается надёжно, ты отдаёшь цикл агенту.

```
  ВООБРАЖЕНИЕ  ──▶    ЗАДАЧА     ──▶   АВТОМАТИЗАЦИЯ
  образ есть      пишешь + проверяешь   отдаёшь цикл
  (он всегда был)   рука об руку с AI      агенту
       └──────────  порядок важен  ──────────┘
```

> Перепрыгнуть к автоматизации, не умея написать и проверить задачу, — значит автоматизировать собственную растерянность. Сначала научись доводить задачу до результата рука об руку с AI; автоматизация — это следующий горизонт, не первый.

</Phase>
