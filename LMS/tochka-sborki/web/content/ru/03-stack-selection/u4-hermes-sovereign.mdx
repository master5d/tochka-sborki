<Phase type="activation">

Вспомни момент, когда один инструмент пытался делать всё сразу — и сломался на полпути.

Не описывай это как «проблема контекстного окна». Представь это как **огород против рынка** 🌱: хозяин огорода сам посеял, сам полил, сам убрал — и знает каждый куст. Покупатель на рынке берёт готовое — быстро, без хлопот. Hermes — это хозяин, который **растит план у себя** и отправляет работников за конкретными результатами, а не доверяет единственному прилавку весь урожай.

Разгляди в голове: какая часть твоего AI-workflow больше всего страдает от того, что агент **делает всё подряд** и теряет нить посреди задачи?

</Phase>

<Phase type="reflection">

Представь театрального режиссёра 🎭 на репетиции. Он не играет за всю труппу по очереди — он держит **картину спектакля целиком** и говорит каждому актёру ровно столько, чтобы тот сыграл свою сцену. Актёры не знают финала — режиссёр знает.

Поймай ту же логику здесь:

> «Я попросил Claude Code рефакторить auth-модуль. На 5-м файле он начал применять стиль из бекенда к фронтенду, забыл про мой первоначальный план, и потратил 200K токенов чтобы вернуться к началу.»

Это паттерн **context-saturation**: актёр попытался сыграть все роли сразу. Hermes держит **только план и состояние** — роль режиссёра:
1. Формулирует подзадачу
2. Запускает sub-agent (Aider в worktree, или Claude Code) — «выход актёра на сцену»
3. Получает результат, проверяет
4. Обновляет план, следующий выход

Контекст Hermes не съедается реализацией. Контекст worker'ов — атомарный, тщательно скоупленный.

</Phase>

## Установка Hermes (упрощённая версия)

> 💡 Полный SOVERN stack — отдельный проект на 1–2 недели. Здесь — **минимальная Hermes-experience** для понимания pattern'а.

### Требования
- Linux/WSL2/macOS
- Python 3.11+
- LiteLLM gateway уже запущен (см. u3, шаги 1–3)

### Скачать Hermes

Hermes — open-source, живёт в репозитории SOVERN:

```bash
git clone https://github.com/sovern-stack/hermes
cd hermes
pip install -e .
```

### Базовый конфиг

`~/.config/hermes/config.yaml`:

```yaml
boss:
  llm: balanced              # alias из LiteLLM
  plan_dir: ~/notes/plans/   # где хранятся планы
  effort: high               # boss всегда думает high

workers:
  refactor:
    type: aider
    llm: fast-pool
    isolation: worktree
  research:
    type: shell
    command: claude -p
    fallback: aider
```

### Первый запуск

```bash
hermes new-plan "Добавить аутентификацию в проект auth-demo"
```

Hermes:
1. Создаст план в `~/notes/plans/auth-demo-2026-05-18.md`
2. Откроет его в редакторе — ты можешь скорректировать
3. На `hermes execute` — начнёт делегировать worker'ам

### Skill system (опционально)

Hermes использует DeepVista Skill Schema — модульные `.md` файлы с YAML frontmatter:

```yaml
---
name: refactor-typescript
type: stateful
trigger: "refactor TypeScript"
worker: refactor
dry_run: true
---
# Refactor TypeScript

При запуске skill'а:
1. Выполнить dry-run всех правок
2. Показать diff, спросить подтверждение
3. После approve — применить
```

Это аналог `.claude/skills/`, но **переносимый между агентами**.
