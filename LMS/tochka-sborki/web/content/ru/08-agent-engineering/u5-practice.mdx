<Phase type="activation">

Цель этого юнита — собрать всё что ты сделал в M8 в **один документ-спецификацию**.

Не описывай это как «документ который надо заполнить» — представь это как **полководца, разворачивающего кампанию по карте**: он не бежит в атаку сам — он расставляет подразделения, определяет цели, прокладывает линии снабжения. Само сражение — потом. Сначала — план на бумаге.

Прокрути в голове задачу из юнита 1. Представь себя полководцем: твои агенты, инструменты, триггеры — это подразделения на карте. Где они стоят? Какой фланг не прикрыт?

</Phase>

<Phase type="reflection">

Хороший спец агента отвечает на эти вопросы **до** написания кода.

Не представляй это как анкету — представь это как **архитектурный чертёж**: дом не строится пока нет чертежа. Ошибка на бумаге стоит карандаш. Ошибка в фундаменте — снос и пересборка. Архитектор держит в голове всё здание сразу — нагрузки, коммуникации, аварийные выходы — ещё до того как забита первая свая.

| Вопрос | Что в ответе |
|---|---|
| **Trigger** | Что запускает агента? (cron / webhook / событие / ручной запуск) |
| **Inputs** | Что приходит? (формат, объём, источник) |
| **Steps** | Из каких узлов состоит pipeline? |
| **AI vs Code** | Какие узлы LLM, какие — код (см. u2)? |
| **Outputs** | Куда пишется результат? |
| **Failure modes** | Что может сломаться, что делать? |
| **Observability** | Где смотреть что произошло? |
| **Cost ceiling** | Сколько максимум денег в день? |
| **Human-in-loop** | Где нужна твоя проверка? |

Если хоть один вопрос без ответа — рано закладывать фундамент. Какой несущий элемент в твоём чертеже пока без опоры?

</Phase>

<Phase type="concept">

**Шаблон спецификации агента**

```markdown
# Agent: [имя]

## Цель
[Одно предложение: что делает агент, какую боль решает]

## Trigger
- Тип: cron / webhook / event / manual
- Параметры: [например, каждые 5 минут / при метке Gmail / при пуше в репо]

## Pipeline
1. [Тип узла] — [что делает] — [in → out]
2. [Тип узла] — [что делает] — [in → out]
3. ...

## AI узлы
Список узлов где используется LLM:
- Узел X: модель [provider/model], промпт в `prompts/x.md`
- ...

## Tools / APIs
- Gmail API (scope: ...)
- Notion API (workspace: ...)
- ...

## Storage
- Куда пишет результат
- Идемпотентность: как избегаем повторов

## Observability
- Trace: Langfuse / OTel / stdout логи
- Alerting: куда уходит ошибка (Telegram / email / Slack)

## Failure modes
| Сценарий | Поведение |
|---|---|
| LLM 500 | Retry 3× с exp backoff, потом fallback модель |
| Bad input | Лог + Telegram alert + skip |
| Storage full | Hard fail + alert |

## Cost
- LLM calls в день: ~N
- Estimated $: ~$X
- Hard cap: $Y (через провайдер бюджеты)

## Human-in-loop
- Где: [шаг N]
- Что подтверждаешь
- Как: [Telegram сообщение / Notion task / email]
```

**Реальный пример заполненной спеки** — упрощённая версия ENERV ingest:

```markdown
# Agent: ENERV Gmail Ingest

## Цель
Письма с тегом `AI Ingest` автоматически попадают в правильную папку
Obsidian-vault как .md файлы.

## Trigger
- Тип: poll
- Параметры: Gmail API каждую минуту, фильтр `label:AI Ingest`

## Pipeline
1. Gmail Trigger — забрать письма — () → [emails]
2. Normalize Input — стандартизация полей — [email] → [normalized]
3. Gemini Router — классификация — [normalized] → [folder, filename]
4. Parse Response — валидация JSON — [llm_out] → [structured]
5. ACL Transformer — генерация .md — [normalized + structured] → [md]
6. Source Check — дедуп — [md] → [md или skip]
7. Write — запись файла — [md] → /home/node/obsidian/{folder}/{filename}.md
8. Mark Ingested — пометка Gmail — [email_id] → ()
9. Remove Ingest Label — снятие триггера — [email_id] → ()

## AI узлы
- Gemini Router — gemini-flash, промпт `gemini-router.md`
- ACL Transformer — gemini-flash, промпт `acl-transformer.md`

## Tools
- Gmail API (modify scope)
- File system (volume mount /home/node/obsidian)

## Storage
- Obsidian vault на host volume
- Idempotency: labels `AI Ingest` снимается, добавляется `AI-Ingest-Done`

## Observability
- n8n executions log
- Langfuse traces для обоих LLM-узлов

## Failure modes
| Сценарий | Поведение |
|---|---|
| Gemini 500 | n8n retry 3× с exp backoff |
| Битый JSON | Лог + письмо оставляется с `AI Ingest` |
| Disk full | Email alert (Resend) |

## Cost
- ~10 писем в день × 2 LLM-вызова = 20 calls/day
- Gemini Flash: ~$0.001/call → ~$0.02/day
- Hard cap: $5/month через Vertex AI budget

## Human-in-loop
- Письма помеченные `AI-Ingest-Review` — раз в неделю просматриваю папку
  Atlas/Notes/Sources вручную, чтобы поймать misclassification
```

Это **на бумаге**. Кода ещё нет. Но любой человек — включая меня самого через полгода — может это прочитать и понять что делает агент.

---

### Вторая половина клона: устав

Спека выше — это **водопровод**: триггеры, узлы, хранилище. Руки агента. Но если твой клон **разговаривает** — отвечает, со-мыслит, ведёт диалог, — одного водопровода мало. Ему нужен **устав**: кто он, во что верит, чего никогда не делает. Это его мозг.

Разница простая: спека отвечает на «как он работает», устав — на «кто он, когда говорит с человеком». Промпт без устава — одноразовый. Устав превращает его в **стабильного со-мыслящего напарника**: завтра, через месяц, в чужих руках он держит ту же рамку.

Шаблон — семь блоков. Заполняется один раз, живёт в `CLAUDE.md` проекта или в системном промпте агента:

```markdown
# Agent Charter: [имя клона]

## Identity — кто он
Одно предложение: роль + стойка. Не «бот для X», а со-мыслящий [роль], который [как держится].
> Пример: «Со-мыслящий редактор: режет лишнее и защищает голос автора, а не переписывает за него.»

## Profile — что он несёт
Контекст, который держит всегда:
- Домен: [сфера]
- С кем работает: [аудитория / ты сам]
- Что про человека важно знать: [ситуация, уровень, запрос]

## Principles — во что верит
3–5 рабочих убеждений, по которым принимает решения:
- [принцип 1]
- [принцип 2]
- [принцип 3]

## Use / Avoid — границы
- Звать, когда: [где он силён]
- Не звать, когда: [где навредит или это не его роль]

## Loop — как работает за один ход
Цикл, который повторяет каждый раз:
1. [сначала уточняет / спрашивает …]
2. [потом …]
3. [в конце …]
> Один фокус за ход. Смысл и решение остаются за человеком.

## Laws — жёсткие рамки
То, что нельзя нарушать:
- Никогда: [запрет]
- Всегда: [обязательство]

## Goal — зачем он есть
Одно предложение: какой результат у человека, если агент работает хорошо.
```

Семь блоков не случайны: **Identity + Profile + Goal** дают агенту опору («кто я, для кого, ради чего»), **Principles + Laws** — характер и тормоза, **Use/Avoid + Loop** — как он действует в моменте. Убери любой — и агент начинает плыть.

</Phase>

<Phase type="practice">

Заполни шаблон спеки выше для **твоей** задачи из юнита 1.

Ограничения:
- Не пиши «как будто я уже всё знаю». Если не знаешь — напиши `[?]` и оставь вопрос.
- Один файл `my-experiments/my-agent-spec.md`
- Минимум 1 страница, максимум 3
- Покажи кому-то (партнёру, другу, ChatGPT) и спроси: «понятно что делает агент?»

**Если твой клон разговаривает** (отвечает, со-мыслит, а не просто гоняет pipeline) — заполни ещё и **устав** по семи блокам выше. Один файл `my-experiments/my-agent-charter.md`. Готовый шаблон лежит в `my-templates/agent-charter.md` — скопируй и заполни под себя.

**Sanity check перед концом юнита:**

- [ ] Trigger четкий
- [ ] Pipeline 3+ шагов
- [ ] Хотя бы 1 LLM-узел и 1 не-LLM-узел
- [ ] Описан хотя бы 1 failure mode
- [ ] Понятно сколько ~стоит в день
- [ ] Указано где смотреть трейсы

Если все 6 галочек — у тебя есть спецификация по которой можно начинать строить.
