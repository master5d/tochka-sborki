# my-templates 🎯

Это твоя библиотека проверенных промптов и шаблонов. Храни здесь то, что **работает**.

---

## 🏗️ Как строить свою библиотеку — 4 шага

Методика построения персональной библиотеки промптов. Пройди её один раз на старте — дальше только пополняй.

### Шаг 1 — Identify Common Tasks

Выпиши 10–15 задач, которые ты делаешь **чаще всего**. Не "написать код", а конкретные:
- "Написать email ответ клиенту с объяснением задержки"
- "Превратить заметки со встречи в action items"
- "Переписать сырой draft поста в LinkedIn-версию"
- "Найти баги в PR перед review"
- "Сгенерировать unit-тесты для нового модуля"

> 💡 Это упражнение — первый час вложений. Дальше оно окупает каждый день.

### Шаг 2 — Create Base Prompts

Для каждой задачи напиши **один** рабочий промпт. Не идеальный — просто рабочий. Используй формулу Meeting 3:

```
Context + Task + Instructions + Data
```

или промежуточный Markdown-шаблон:

```
Role / Objective / Context / Instructions / Notes
```

### Шаг 3 — Organize

Сгруппируй промпты по категориям — так ты за 5 секунд найдёшь нужный:

```
prompts/
├── content/       — посты, emails, drafts
├── code/          — generation, refactor, tests
├── analysis/      — research, audits, reviews
└── ops/           — scheduling, summaries, routines
```

Имена файлов — ответ на вопрос "когда я это использую?": `prompt-email-client-delay.md`, `prompt-linkedin-post-from-notes.md`.

### Шаг 4 — Refine через прошедшие упражнения

Дальше — эволюция. Каждый раз, когда промпт даёт **отличный** результат в `my-experiments/` — переноси его сюда с новым номером версии. Каждый раз, когда он даёт плохой — фиксируй что именно не сработало и правь.

## Как использовать

### 1. Стартовый набор

Скопируй промпт из нужного файла, подставь конкретные значения, готово!

```bash
# Пример: берёшь prompt-backend.md
# Подставляешь свой технический стек
# Даёшь агенту
# Получаешь результат
```

### 2. Создай свой шаблон

После успешного эксперимента (из `my-experiments/`) переноси лучшее в шаблоны:

```bash
# В my-experiments:
experiment-new-prompt-style.md → ОТЛИЧНО!

# В my-templates:
prompt-my-style.md → ДОБАВЛЯЮ
```

### 3. Версионирование

Если шаблон эволюционирует, добавляй версию:

```
prompt-backend-v1.md     # Старая версия (архив)
prompt-backend-v2.md     # Текущая версия
prompt-backend.md        # Символическая ссылка на v2
```

## Формат файла с промптом

```markdown

## Когда использовать
[Для какого типа задач?]

## Примеры задач
- Задача 1
- Задача 2

## Промпт

### Роль
Ты - ...

### Задача
...

### Input
...

### Output
...

### Constraints
...

## Результаты
[Насколько хорошо это работает?]

## Примечания
[Что нужно помнить при использовании?]

## Последний раз обновлен
[Дата]
```

## Примеры

### Пример 1: Базовый промпт для backend

```markdown
# Backend API Development

## Когда использовать
Когда нужно написать новый API endpoint или модуль на backend

## Промпт

### Роль
Ты - опытный backend разработчик с опытом в [твой стек].

### Задача
Создай [что нужно создать]

### Input
- Данные: [формат данных]
- Требования: [список требований]

### Output
- Файл [название].ts
- Типизация для TypeScript
- Примеры использования

### Constraints
- Максимум [строк] кода
- Используй [библиотека]
- Избегай [что избегать]

## Результаты
⭐⭐⭐⭐⭐ - Отличные результаты с первого раза

## Примечания
Агент отлично понимает требования с этим стилем промпта.
```

## 🎯 Правила библиотеки

1. **Только проверенное** - добавляй только то, что реально работает
2. **Актуальность** - обновляй шаблоны по мере эволюции своего стиля
3. **Документирование** - описывай, когда и как использовать
4. **Версионирование** - храни старые версии для истории
5. **Переиспользование** - не копируй, наследуй (используй как базу)
